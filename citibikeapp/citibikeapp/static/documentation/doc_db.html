Breaks (Manage Default Breaks)

The default schedule for breaks. The table has 4 columns: vID int, tType int, publishTime text, publishDateTime text. A vehicle’s default breaks are deleted when the vehicle signs out.

 

vID is an integer representing the vehicle the break is assigned to. Ex. 10

 

tType is an integer representing which type of break is assigned. It should only be 16, 17 or 18 because only these are break task types. The correct sequence of scheduled breaks is 17, 16, then 18 (short break, lunch, short break).  Ex. 16

 

publishTime is a string in format hh:mm:ss representing when the breaks should be assigned by the algorithm. Ex. 15:30:00

publishDateTime is a string in format YYYY-MM-DD hh:mm:ss representing when the breaks should be assigned by the algorithm. Ex. 2017-04-23 15:30:00

ClosedForbiddenStations

An archive for the deleted recurring forbidden station events. The table has 8 columns: fsID int, sID int, sComment text, startTime text, endTime text, repeat text, publishTime text, closeTime text

 

fsID is a unique integer used to identify the forbidden event. Ex. 1

 

sID is an integer used to represent the station. Ex. 72

 

sComment is a string representing the comment. Ex. Test station

 

startTime is a string in format hh:mm:ss representing when the station should start to be forbidden. Ex. 15:30:00

 

endTime is a string in format hh:mm:ss representing when the station should stop being forbidden. Ex. 15:30:00

 

repeat is a string representing which days the station is forbidden. 12345 – weekdays, 67 – weekends, 1234567-everyday.

 

publishTime is a string in format YYYY-MM-DD hh:mm:ss representing when the forbidden station event is created. Ex. 2017-04-23 13:30:00

 

closeTime is a string in format YYYY-MM-DD hh:mm:ss representing when the forbidden station event is deleted. Ex. 2017-04-23 13:30:00

ClosedTasks

An archive for all tasks that are completed or rejected.

There are 21 columns in the table, including most columns in OpenTasks table: : tID int, publishTime text, acceptTime text, vID int, dID1 text, dID2 text, requirement text, completionTime text, priority text,  tType int, sID int, bikeNum int, comment text, estComplete text, arrivalTime text, rejTime text, reasonID int

; and 4 additional columns: status int, closedTime text, actualDuration text, actBikeNum int.

 

Columns in OpenTasks but NOT in ClosedTasks: orderNum int, pDL int, fixTask int

 

status is an integer representing whether the task is completed or rejected. Ex. 1 for completed, 0 for rejected.

 

closedTime is a string in format YYY-MM-DD hh:mm:ss representing the time when the task is completed or rejected. Ex. 2017-06-08 13:05:11

 

actualDuration is not used yet.

 

actBikeNum is an integer representing how many bikes the rebalancer actually moved. Ex. 5

DeletedTasks

An archive for all tasks that are manually deleted.

There are 18 columns in the table, including most columns in OpenTasks table: : tID int, publishTime text, acceptTime text, vID int, dID1 text, dID2 text, requirement text, completionTime text, priority text,  tType int, sID int, bikeNum int, comment text, estComplete text, arrivalTime text, rejTime text, reasonID int; and 1 additional columns: deletedTime text,

Columns in OpenTasks but NOT in DeletedTasks: orderNum int, pDL int, fixTask int

 

deletedTime is a string in format YYY-MM-DD hh:mm:ss representing when the task is deleted. Ex. 2017-06-08 13:05:11

DriversShift

A table that keep tracks of the shifts information of all drivers. There are 10 columns: dID text, vID int, signInTime text, signOutTime text, lunchCount int, breakCount int, signInTimeEstimate text, signOutTimeEstimate text, comment text, disproved int

 

dID is a unique string used to identify each driver. Ex. cd243

 

VID is an integer represents which vehicle the driver is on most recently. It is only updated when the driver starts shift. Ex. 10

 

signInTime is a string in format YYYY-MM-DD hh:mm:ss representing when the driver starts shift. It is reset to null when the driver’s request to end shift is confirmed. If it is not null, the system recognizes the driver as in the shift. Ex. 2017-06-08 13:05:11

 

signOutReqTime is a string in format YYYY-MM-DD hh:mm:ss representing when the driver requires to end shift. It is reset to null when the driver’s request to end shift is confirmed. If it is not null, the system recognizes the driver requesting to end shift. Ex. 2017-06-08 13:05:11

 

lunchCount is an integer representing how many lunch breaks the driver has taken in this shift. It is reset to 0 when the driver’s request to end shift is confirmed. Ex. 1

 

breakCount is an integer representing how many short breaks the driver has taken in this shift. It is reset to 0 when the driver’s request to end shift is confirmed. Ex. 1

 

signInTimeEstimate is a string in format hh:mm:ss representing when the the driver should start the shift. It can be updated by the dispatcher. Ex. 13:05:11

 

signOutTimeEstimate is a string in format hh:mm:ss representing when the the driver should end the shift. It can be updated by the dispatcher. Ex. 13:05:11

 

comment is a string representing the comment. Ex. test

ForbiddenStationsTemp

All non-recurring forbidden station events. An entry is added when an opentask is deleted.

 The table has 8 columns: fsID int, sID int, sComment text, startTime text, endTime text, publishTime text, startD text, endD text

 

fsID is a unique integer used to identify the forbidden event. Ex. 1

 

sID is an integer used to represent the station. Ex. 72

 

sComment is a string representing the comment. Defaults to “deleted”. Ex. deleted

 

startTime is a string in format hh:mm:ss representing when the station should start to be forbidden. Usually the time is the same as the second half of publishTime. Ex. 13:30:00

 

endTime is a string in format hh:mm:ss representing when the station should stop being forbidden. Ex. 15:30:00

 

publishTime is a string in format YYYY-MM-DD hh:mm:ss representing when the forbidden station event is created. Ex. 2017-04-23 13:30:00

 

startD is a string in format YYYY-MM-DD hh:mm:ss representing when the forbidden station event starts. Usually it’s the same as publishTime. Ex. 2017-04-23 13:30:00

 

endD is a string in format YYYY-MM-DD hh:mm:ss representing when the forbidden station event ends. Ex. 2017-04-23 13:30:00

 

OpenTasks

All tasks that are assigned but not completed or rejected. Some tasks might have been accepted.

There are 20 columns in the table: tID int, publishTime text, acceptTime text, vID int, dID1 text, dID2 text, requirement text, completionTime text, priority text,  tType int, sID int, bikeNum int, comment text, estComplete text, orderNum int, pDL int, arrivalTime text, fixTask int, rejTime text, reasonID int

 

tID is a unique integer used to identity each task. Ex. 1

 

publishTime is a string in format YYYY-MM-DD hh:mm:ss representing the time when the task is generated wither manually or by algorithm. Ex. 2017-06-14 08:08:50

 

acceptTime is a string representing the time when the task is accepted. Ex. 2017-06-14 08:08:50

 

vID is an integer representing the vehicle the task is assigned to. Ex. 1

 

dID1 is a string representing the driver whom the task is assigned to. Ex.qw24

 

dID2 is a string representing the second driver whom the task is assigned to. Ex.qw24

 

requirement is a string representing the restrictions of which drivers are able to do the task. (is not used)

 

completionTime is a string in format YYYY-MM-DD hh:mm:ss representing the deadline when the task should be completed. Ex. 2017-06-14 08:08:50

 

priority is an integer (in string format) representing the priority of the task. 1 for LOW, 5 for MIDDLE, 7 for HIGH. Ex. 1

 

tType is an integer representing the type of the task, it corresponds to the tType in the Tasks table. ex . 1 (which corresponds to pick up bike in the Tasks table)

 

sID is an integer representing the station where the task should be done, it corresponds to the sID in the Stations table. Ex. 79

 

bikeNum is an integer representing the number of bikes the task involve. Ex. 4

 

comment is a string that includes any additional comment of the task.

 

estComplete is a string in format YYYY-MM-DD hh:mm:ss representing when the task is estimated to be completed. Ex. 2017-06-14 08:08:50

 

orderNum is an integer representing the order of the task for each vehicle. Ex. 2

 

pDL is an integer representing whether the task has passed deadline. 0 for not, 1 for yes. If a task has pDL 1, it is colored red in the web page. If a task’s completionTime is updated, pDL reset to 0. pDL field defaults to 0. Ex. 0

 

arrivalTime is a string in format YYYY-MM-DD hh:mm:ss representing when the driver has arrived for the task. If this is not empty, the system thinks the driver has arrived and will reveal the bike number to the driver in the iOS app. Ex. 2017-06-14 08:08:50

 

fixTask is an integer representing whether the task is in the prefix task list. 0 for no, 1 for yes. Only task in the prefix task list can be reordered or updated. Ex. 1

 

rejTime is a string in format YYYY-MM-DD hh:mm:ss representing when the driver has requested to reject the task. If this is not empty, the system thinks the driver has made a request and will ask the dispatcher to approve or disprove the request. Ex. 2017-06-14 08:08:50

 

reasonID is an integer representing the reason why the driver request a rejection. The corresponding meaning is in the ReasonCode table. Ex. 1

ShiftHist

An archive used to track the sign-in and sign-out of drivers.

There are 7 columns in the table: dID text, vID int, startTime text, endTime text, comment text, lunchCount int, breakCount int.

 

dID: a string representing the id of the driver. Ex. qq123

 

vID: an integer representing the id of the vehicle that the driver is on. Ex. 10

 

startTime: a string in format YYYY-MM-DD hh:mm:ss representing when the shift starts. Ex. 2017-06-08 13:05:11

 

endTime: a string in format YYYY-MM-DD hh:mm:ss representing when the shift ends. If it is null, the driver may still be on the shift or there was a problem with ending shift process. Ex. 2017-06-08 13:05:11

 

comment: a string representing the comment for the shift. (not used now)

 

lunchCount: an integer representing how many lunch breaks the driver takes during this shift. Defaults to null. Only updated when the shift end request is confirmed. Ex. 1

 

breakCount: an integer representing how many short breaks the driver takes during this shift. Defaults to null. Only updated when the shift end request is confirmed. Ex. 1

Stations

All stations in https://feeds.citibikenyc.com/stations/stations.json . All information in the json is included in the table.

The table has 19 columns: sID int, stationName text, availableDocks int, totalDocks int,latitude real, longitude real, statusValue text, statusKey int, availableBikes int, stAddress1 text ,stAddress2 text, city text, postalCode int, location text, altitude real, testStation numeric, LCT text, landMark text, executionTime text.

 

(if not specified, the column name and the json key are the same)

 

The table is only supposed to be updated/changed by querying the json above.

 

sID corresponds to key “id” in the json. It’s a unique integer used to identify each station.  Ex. 79

 

stationName is a string representing the name of the station. (According to my observation, the same as stAddress1) Ex. W 52 St & 11 Ave

 

availableDocks is an integer representing the empty docks at the stations. Ex. 15

 

totalDocks is an integer representing the capacity of the station. Ex. 38

 

latitude is a floating number representing the latitude of the station. Ex. 40.767272

 

longitude is a floating number representing the longitude of the station. Ex. -73.993929

 

statuValue is a String representing the station’s status. Ex. “In Service” or “Not In Service”

 

statusKey is an integer representing the station’s status. Ex.1 for  “In Service”, 3 for  “Not In Service”

 

availableBikes is an integer representing the bikes at the station. Ex. 18

 

stAddress1 is a string representing the address of the station. (According to my observation, the same as stationName)  Ex. W 52 St & 11 Ave

 

LTC corresponds to key “Last Communication Time” in the json. It represents the most recent time the json communicates with the station. Ex. 2017-06-08 11:03:59 AM

 

executionTime is the time when the json is queried. It is in 24-hour format: yyyy-MM-dd HH:mm:ss. Ex. 2017-06-08 13:05:11

 

stAddress2, city, postalCode, location, altitude & landMark are all empty by now.

 

testStations are all 0.                                          

Tasks

All types of task. The table has 3 columns: tType int, tName text, deltaBike int.

 

tType is a unique integer used to identify each task. It increments automatically if new task types are added. Ex. 1

 

tName is a string that represents the name of each task. It is determined by the user when adding new task types. Ex. Pick up bike

 

deltaBike is either -1,1, or 0. It is used to represent whether the number of bikes on the vehicle increase or decrease after completing this type of task. 1 means increasing and -1 means decreasing. Ex. -1

Vehicles

All vehicles.

                There are 23 columns: vID int ,vName text, dID1 text, dID2 text, capacity int, vBike int, tID int, vACLatitude real, vACLongtitude real, vNXsID int, vNXLatitude real, vNXLongtitude real, vSLatitude real, vSLongtitude real, vELatitude real, vELongtitude real, vSsID int, vEsID int, vACsID int, startTime text, endTime text, vBikeBroken int, LFTime text

 

vID is an unique integer used to identify each vehicle. Ex. 1

 

vName is a string representing the name of the vehicle. Ex. Truck1

 

dID1 is a string representing the id of the first driver of the vehicle. Ex. qq123

 

dID2 is a string representing the id of the second driver of the vehicle. Ex. ww123

 

capacity is an integer representing how many bikes the truck can carry max. Ex. 30

 

vBike is an integer representing how many bikes are on the truck now. Ex. 10

 

tID is an integer representing the last accepted task. Ex. 32

 

vACLatitude is a float number representing the latitude of the location of the truck where the drivers did the most recent communication with the server (update vehicle info/ accept, reject, arrive or complete a task).  Ex. 40.767272

 

vACLongtitude is a float number representing the longitude of the location of the truck where the drivers did the most recent communication with the server (update vehicle info/ accept, reject, arrive or complete a task). Ex. -73.993929

 

vNXsID is an integer representing the ID of the station where the vehicle’s current task will happen. Ex. 72

 

vNXLatitude is a float number representing the latitude of the location where the vehicle should be going to. Usually it’s the latitude of vNXsID.  Ex. 40.767272

 

vNXLongtitude is a float number representing the longitude of the location  where the vehicle should be going to. Usually it’s the longitude of vNXsID.  Ex. -73.993929

 

vSLatitude is a float number representing the latitude of the starting location of the truck.  Ex. 40.767272

 

vSLongtitude is a float number representing the longitude of the starting location of the truck. Ex. -73.993929

 

vELatitude is a float number representing the latitude of the ending location of the truck.  Ex. 40.767272

 

vELongtitude is a float number representing the longitude of the ending location of the truck. Ex. -73.993929

 

vSsID is an integer representing the ID of the station where the vehicle starts the shift. Ex. 72

 

vEsID is an integer representing the ID of the station where the vehicle’s shift ends. Ex. 72

 

vACsID is an integer representing the ID of the station where the vehicle complete the last task. It is passed to the algorithm as the current location of the vehicle. Ex. 72

 

startTime is a string in format YYYY-MM-DD hh:mm:ss representing when the shift starts. Ex. 2017-06-08 13:05:11

 

endTime is a string in format hh:mm:ss representing when the shift ends. Ex. 13:05:11

 

vBikeBroken is an integer representing how many broken bikes are on the vehicle. Ex. 4

 

LFTime is a string in format YYYY-MM-DD hh:mm:ss representing when vehicle completes the previous . Ex. 2017-06-08 13:05:11
